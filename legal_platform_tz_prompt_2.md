## 0. Преамбула для агента

Ты — старший fullstack-разработчик и продуктовый архитектор. Тебе поручено спроектировать и реализовать **внутреннюю корпоративную LegalTech-платформу** для юридического департамента крупной производственной компании (заводы, сбыт, закупки, дочерние общества, тысячи договоров в год, претензионно-исковая работа, корпоративные процедуры).

Продукт собирается «из лучших практик рынка»: вдохновляйся **Ironclad, ContractPodAi (Leah), Sirion, Agiloft, Litify, Onit, LegalOn, Spellbook, Harvey, LinkSquares, Brightflag, Doczilla, ПравоТех Casebook, XSUD, Юрайт**, но не копируй ни один из них целиком. Бери лучшее.

**Жёсткие правила:**

1. **Никакой геймификации**, никаких «бейджиков», «достижений», «уровней», конфетти, эмодзи в продуктовой части.
2. **Никаких «фишек ради фишек»**: каждая кнопка, поле, экран должны иметь измеримую пользу для юриста или внутреннего заказчика.
3. **Минимализм + плотность данных**: интерфейс корпоративный, тихий, читается часами без усталости. Эстетика на уровне Linear / Notion / Stripe Dashboard / Vercel — но более «деловая» и сдержанная.
4. **Скорость > красота**. Каждое действие должно занимать минимум кликов. Идеал: 2–3 клика на типовое действие.
5. **AI там, где он реально экономит часы**, а не как маркетинг. Если AI не повышает качество — не вставляй его.
6. **Принцип «одного окна»**: юрист и бизнес-заказчик не должны выходить из системы для рутины.
7. **Без захардкоженной геймификации, без «ассистент пишет ободряющие сообщения», без аватарок-эмоджи.**

Если тебе кажется, что какое-то решение «ради вау-эффекта», — выкидывай его.

## 1. Контекст компании и пользователи

### 1.1 Компания

- Крупная производственная (промышленная) группа компаний.
- Холдинговая структура: головная компания + дочерние общества + региональные филиалы.
- Объёмы: ~5 000–20 000 договоров в год, сотни активных судебных дел, сотни доверенностей, постоянные закупки (44-ФЗ / 223-ФЗ или коммерческие), тендеры, корпоративные процедуры (ОСА, СД, M&A, реорганизации), претензионная работа с поставщиками и подрядчиками, постоянные изменения в законодательстве.
- Юристов: 30–150 человек, разнесены по регионам и дочкам.

### 1.2 Роли пользователей (RBAC)

Спроектируй ролевую модель, которая поддерживает как минимум:

1. **Бизнес-заказчик (инициатор)** — закупщик, коммерсант, кадровик, инженер. Не юрист. Подаёт заявку, отвечает на уточнения, видит статус.
2. **Юрист-исполнитель** — обрабатывает заявку, готовит/проверяет документ, ведёт дело.
3. **Старший юрист / руководитель направления** — согласует, эскалирует, распределяет нагрузку.
4. **Руководитель юридического департамента (CLO / Директор по правовым вопросам)** — видит дашборды, аналитику, нагрузку, риски, бюджет.
5. **Внешний консультант / адвокат** — ограниченный доступ к конкретным делам через защищённый портал.
6. **Администратор системы** — управляет шаблонами, плейбуками, справочниками, пользователями.
7. **Аудитор / комплаенс** — read-only + доступ к журналу действий.

Каждая роль видит **свой стартовый дашборд**, заточенный под её задачи (role-based dashboard, как у SAP Fiori и Salesforce).

### 1.3 Существующий портал «Право» и набор уже работающих инструментов

В компании **уже работает** корпоративный портал юридического департамента под названием **«Право»** (заголовок страницы — «Право», подзаголовок — «Корпоративный портал юридического департамента»). На главной странице — витрина из 12 проектов-утилит, к которым пользователи привыкли. Эти утилиты **не выбрасываются** — они становятся фундаментом новой платформы, мигрируют в неё с сохранением логики, URL-структуры (где возможно) и привычного юристам поведения.

**Текущий визуальный стиль главной витрины (сохранить как канон):**

- Светло-голубой/серый фон страницы.
- Карточки в светлых тонах с лёгкой тенью, скруглением ~8px.
- В верхней части страницы — название «Право» с маленькой иконкой весов слева, под ним — подзаголовок «Корпоративный портал юридического департамента».
- Тег-чип под заголовком: «Направление: Право · Проектов: N».
- Кнопка `+ Добавить проект` справа сверху, тёмно-синий фон, белый текст, скругление.
- Карточка каждого проекта: иконка (одна тематическая, плоская, ~32–40px), заголовок (16–18px, semibold), 2–3 строки описания (14px, серый текст), снизу — ссылка `Открыть →` красным цветом.
- Сетка 4 колонки на десктопе, адаптивная вниз до 1 колонки на мобильном.
- Никаких лишних блоков, баннеров, рекламы фич. Только сетка проектов и шапка.

**Эту витрину сохранить как стартовый экран** для всех пользователей, разместив её на роуте `/` (или `/tools`). Дашборды по роли (см. раздел 4.10 ниже) — отдельный экран на `/dashboard`. Пользователь сам выбирает в настройках, что показывать на старте: витрину инструментов или дашборд.

У бизнес-заказчика и обычного юриста этой кнопки нет.

## 2. Технологический стек (рекомендация, можешь уточнить)

- **Frontend:** React + TypeScript, Next.js (App Router), TailwindCSS, shadcn/ui, Radix primitives, TanStack Table, TanStack Query, Recharts/Visx для графиков, Zustand или Jotai для локального стейта, react-hook-form + zod.
- **Backend:** Node.js (NestJS) или Python (FastAPI). Выбери один и обоснуй в README.
- **БД:** PostgreSQL 16 (основная), Redis (кэш, очереди), S3-совместимое хранилище для файлов (MinIO для self-hosted), OpenSearch/Elasticsearch для полнотекстового поиска по документам.
- **Очереди и фоновые задачи:** BullMQ (Node) или Celery (Python).
- **AI-слой:** абстракция над LLM-провайдером (Anthropic Claude / OpenAI / локальная LLM через vLLM/Ollama). По умолчанию — Anthropic Claude через `@anthropic-ai/sdk`. Embeddings — локальные (bge-m3, e5-multilingual) для приватности.
- **Vector DB:** pgvector (внутри Postgres, чтобы не плодить инфраструктуру) или Qdrant.
- **Auth:** Keycloak / Authentik (SSO, SAML, OIDC), интеграция с корпоративным AD/LDAP. MFA обязательно.
- **Документы:** docx — `docxtemplater` + `docx` (Node), `python-docx` (Python). PDF — `pdf-lib`, рендер из docx через LibreOffice headless. Электронная подпись — интеграция с КриптоПро / SignMe / DocuSign (опционально).
- **OCR:** Tesseract 5 + предобработка через OpenCV. Для русских сканов — `tesseract --lang rus`.
- **Контейнеризация:** Docker + docker-compose для разработки, Helm-чарт для прода (Kubernetes).
- **Наблюдаемость:** OpenTelemetry, Loki + Grafana, Sentry.
- **Тестирование:** Vitest/Jest, Playwright для E2E, k6 для нагрузочных.

Структура монорепо: `apps/web`, `apps/api`, `apps/worker`, `packages/ui`, `packages/shared-types`, `packages/ai-core`, `infra/`.

## 3. Дизайн-система и UX-принципы

### 3.1 Визуальная философия

- **Светлая тема + опциональная тёмная.** Светлая по умолчанию.
- **Нейтральная палитра:** базовый фон `#FAFAFA` / `#FFFFFF`, поверхности `#F4F4F5`, текст `#0A0A0A` / `#52525B`, бордеры `#E4E4E7`. Акцент — один сдержанный цвет (например, глубокий индиго `#4338CA` или корпоративный, заменяемый CSS-переменной).
- **Семантика цвета строгая:** зелёный — успех/согласовано, янтарный — внимание/дедлайн близко, красный — риск/просрочено, серый — нейтрально/черновик. Никаких других цветов в продуктивной части.
- **Шрифт:** Inter или системный SF/Segoe. Для документов — Pt Serif / Charter.
- **Размер базы:** 14px, межстрочный 1.5. Заголовки H1 24px, H2 18px, H3 16px. Никаких декоративных шрифтов.
- **Скругления 6px**, тени минимальные (`shadow-sm`), 1px бордеры. Никаких «glassmorphism», «neumorphism», градиентов.
- **Иконки:** `lucide-react`, единый штриховой стиль 1.5px.
- **Иллюстрации:** только функциональные пустые состояния, монохром.

### 3.2 Информационная плотность

- Таблицы — основной носитель данных. Строки 36–40px, не «воздушные» 80px. Sticky header, sticky первая колонка, виртуализация (TanStack Virtual).
- Каждая таблица: фильтры сверху, сохранение представлений (saved views), быстрый поиск, экспорт CSV/XLSX, групповые действия.
- **Прогрессивное раскрытие**: на первом экране — самое важное, детали по клику в боковую панель (drawer), а не в новый роут, чтобы не терять контекст.
- Дашборды — карточки с одной KPI, без «10 виджетов на одном экране без иерархии».

### 3.3 Навигация

- Левый sidebar, сворачиваемый. Структура верхнего уровня:
  - Главная (дашборд по роли)
  - Заявки (Intake)
  - Договоры (CLM)
  - Документы и шаблоны
  - Дела (matters: суды, претензии, корпоративные)
  - Контрагенты
  - Доверенности
  - Корпоративное право
  - Compliance
  - База знаний
  - Аналитика
  - Администрирование (только для админов)
- Глобальный command palette (`Ctrl+K`) — мгновенный переход к любой сущности по номеру/названию/контрагенту. Это **обязательно**, как в Linear.
- Хлебные крошки на каждом экране глубже 2-го уровня.
- Уведомления — бесшумный значок, без всплывашек кроме критичных дедлайнов.

### 3.4 Микровзаимодействия

- Действия должны давать тихий, понятный фидбек: тост 2.5 сек в правом нижнем углу.
- Никаких confetti, никаких «🎉 Отличная работа!». Просто «Договор согласован» / «Заявка отправлена».
- Optimistic UI везде, где это безопасно.
- Все деструктивные действия — через модалку с подтверждением.
- Горячие клавиши: `c` — создать, `/` — поиск, `g+d` — перейти к договорам, и т.д. Документация в command palette.

### 3.5 Доступность

WCAG 2.1 AA: контраст 4.5:1+, навигация с клавиатуры, ARIA-метки, фокус-индикаторы.

### 4.0 МОДУЛЬ: Витрина инструментов (миграция существующего портала «Право»)

**Зачем:** сохранить уже знакомую юристам стартовую страницу с 12 действующими утилитами и встроить каждую из них в архитектуру новой платформы. Утилиты должны работать **двумя способами**: (а) как самостоятельные мини-страницы из витрины (для разовых задач), (б) как контекстные действия внутри карточек договоров/дел/контрагентов (для встроенного использования).

#### 4.0.1 Главный экран — витрина

- Реализовать страницу `/` (или `/tools`) **точно по визуальному канону раздела 1.3**: шапка «Право» + подзаголовок + чип «Направление: Право · Проектов: N» + сетка карточек 4×3 + кнопка `+ Добавить проект` (только админу).
- Карточки рендерятся из конфига `tools.config.ts` (или из БД-таблицы `tools`), а не хардкодом, чтобы можно было добавлять/убирать инструменты без релиза.
- У каждой карточки: иконка (компонент из `lucide-react` или кастомный SVG), `title`, `description`, `href`, `enabled`, `status` (`live` / `in_development` / `archived`), `roles` (кому видна), `category` (для будущих разделов).
- Утилиты со статусом `in_development` показываются в карточке с пометкой «В разработке» серым шрифтом, ссылка `Открыть →` ведёт на страницу-заглушку с описанием планируемого функционала.
- Поисковая строка над сеткой (опционально, появляется при N > 16 карточек).

#### 4.0.2 Маппинг 12 утилит на модули платформы

Каждая утилита **сохраняется как отдельная страница**, но её код — это тонкая обёртка над сервисами основной платформы. Никакого дублирования бизнес-логики.

| Утилита | Куда встраивается | Доп. использование внутри платформы |
|---------|-------------------|-------------------------------------|
| **1. Автозаполнение** | Модуль 4.2.2 «Конструктор шаблонов». Под капотом — сервис `TemplateFillerService`. | Кнопка `Заполнить из карточки` внутри карточки контрагента и карточки договора. |

#### 4.0.3 Архитектурные требования к утилитам

- **Каждая утилита — отдельный роут**, лениво-загружаемый чанк. На главную витрину не тянем код всех утилит сразу.
- **Общие сервисы** (OCR, обезличивание, расчёт дней, проверки контрагентов и т.д.) выделить в `packages/legal-services/` — они переиспользуются и витриной, и основными модулями.
- **Конфигурация витрины** хранится в БД (`tools` и `tool_categories`), управляется из админки.
- **Аналитика использования каждой утилиты** — обязательно. Какая утилита наиболее востребована, кем, как часто. Эти данные нужны для планирования следующих итераций.
- **Каждая утилита должна работать самостоятельно** даже при недоступности внешних интеграций (graceful degradation). Например, если ФНС API недоступен — Мониторинг показывает кэшированный результат с пометкой «данные на дату X».
- **Все утилиты — двуязычные-ready** (i18n с самого начала, даже если сейчас только русский), так как могут понадобиться англоязычные версии для иностранных консультантов и контрагентов.

#### 4.0.4 Что добавить к существующим утилитам в новой версии

При миграции **не просто перенести как было**, а улучшить там, где это даёт явную пользу:

- **Автозаполнение** → добавить выбор шаблона из библиотеки + предпросмотр результата перед скачиванием.
- **Сравнение версий** → добавить семантический diff (см. 5.9), а не только текстовый. Подсветка изменений по типам: «изменена сумма», «добавлен пункт», «удалена ответственность».
- **Мониторинг** → агрегированный отчёт по контрагенту в одном PDF + автосохранение в историю проверок.
- **Калькулятор пеней** → сохранение расчёта в карточку дела/претензии + экспорт расчёта как Приложение к иску.
- **Юридический помощник** → расширить за пределы трудовых вопросов (закупки, налоги, корпоративное), используя RAG по реальной базе знаний компании.
- **Оценка рисков** → реализовать с нуля по разделу 4.2.3 (AI-ревью с playbook).
- **Оценка дебиторской задолженности** → реализовать по описанию в маппинге выше.

### 4.1 МОДУЛЬ: Единое окно приёма заявок (Legal Intake)

**Зачем:** убрать хаос email-обращений «привет, гляньте договорчик». Любой бизнес-заказчик подаёт структурированную заявку, юрдеп получает контекст с первого раза.

**Сценарии:**

- Закупщик: «Нужно проверить договор поставки от контрагента X».
- HR: «Нужен трудовой договор для нового сотрудника».
- Коммерсант: «Нужна доверенность на представителя».
- Производство: «Нужна юридическая позиция по нештатной ситуации».

**Экраны:**

- **Витрина услуг юрдепа:** карточки с типами обращений (Согласование договора / Подготовка документа / Юридическая консультация / Претензия / Доверенность / Корпоративная процедура / Прочее). Каждая карточка — отдельная динамическая форма с релевантными полями (форма генерируется из конфига, не хардкодом).
- **Форма заявки:** поля адаптируются к типу. Обязательные: название, контрагент (с автоподтягиванием из реестра), сумма, срок, приоритет, прикреплённые файлы. AI-классификатор предлагает тип заявки по тексту описания (опционально).
- **Карточка заявки:** статус (Новая → В работе → Ожидает уточнения → На согласовании → Закрыта), назначенный юрист, SLA-таймер, лента событий, чат с юристом, файлы.
- **Очередь заявок (для юристов):** табличное представление + Kanban-альтернатива, фильтры по типу/SLA/исполнителю/контрагенту, групповое назначение.

**Логика маршрутизации:**

- Конфигурируемые правила: «договор поставки от 10 млн → старший юрист по договорной работе», «трудовой спор → юрист по труду», «корпоративная процедура → корпоративный отдел».
- Автоматическая постановка SLA по типу заявки.
- Эскалация при просрочке SLA.

**Модель данных (минимум):** `requests(id, type, title, description, requester_id, assignee_id, status, priority, sla_deadline, counterparty_id, amount, currency, created_at, updated_at, closed_at, metadata jsonb)`.

### 4.2 МОДУЛЬ: Управление договорами (CLM — Contract Lifecycle Management)

Это **сердце системы**. Покрывает полный жизненный цикл договора: запрос → драфт → согласование → подписание → исполнение → продление/расторжение → архив.

#### 4.2.1 Реестр договоров

- Единый репозиторий всех договоров компании.
- Карточка договора:
  - Реквизиты: номер, дата, тип, стороны (наша + контрагент), сумма, валюта, срок действия, автопродление (yes/no), ответственный со стороны бизнеса, ответственный юрист.
  - Файлы (все версии, с diff между версиями), подписанная версия отдельно.
  - Связанные сущности: дополнительные соглашения, акты, приложения, переписка, претензии, судебные дела.
  - **Реестр обязательств** (extracted obligations): что мы должны (поставить, оплатить, отчитаться) и в какой срок. Каждое обязательство — отдельная сущность с владельцем и дедлайном.
  - Лента событий: все действия по договору с таймстампами и автором.
  - Метаданные: автоматически извлечённые AI ключевые поля (jurisdiction, governing law, termination clause, liability cap, IP rights, confidentiality term, и т.д.).
- Полнотекстовый поиск по содержимому договоров (Elasticsearch + извлечённый из docx/pdf текст).
- Расширенные фильтры: по типу, контрагенту, сумме, сроку, статусу, отделу, региону.
- Saved views: «Мои активные договоры», «Договоры с просроченными обязательствами», «Договоры на продление в ближайшие 60 дней».

#### 4.2.2 Конструктор шаблонов и автогенерация документов

- Юрист-методолог создаёт **интерактивный шаблон**: docx с переменными `{{var}}` + конфиг полей (тип, обязательность, валидация, источник — справочник/ЕГРЮЛ/ручной ввод/расчёт).
- Условные блоки: «Если предоплата > 30%, добавить параграф X».
- Подгрузка реквизитов контрагента из реестра контрагентов одним кликом.
- Подгрузка реквизитов нашей стороны из справочника юрлиц холдинга.
- Подгрузка данных из ЕГРЮЛ/ЕГРИП через API ФНС или Dadata.
- Автонумерация по правилам (год/тип/порядковый).
- Версионирование шаблонов с историей изменений и согласованием изменений (шаблон — тоже документ под контролем).
- Бизнес-заказчик может сам собрать типовой документ через wizard, не дёргая юриста (как Doczilla — 50+ шаблонов на самообслуживании).

#### 4.2.3 AI-проверка договоров (Contract Review / Redlining)

Это **ключевая AI-функция**. Реализовать как central feature.

- **Загрузка договора от контрагента (third-party paper)** в docx или pdf.
- AI выполняет:
  - Извлечение ключевых полей в структурированном виде (стороны, предмет, сумма, срок, ответственность, гарантии, штрафы, jurisdiction, governing law, IP, конфиденциальность, форс-мажор).
  - Сравнение с **playbook** компании — внутренним сводом правил («неустойка не более 0.1%/день, не более 10% от суммы», «срок оплаты не менее 30 дней», «применимое право — РФ», и т.д.). Playbook конфигурируется юристами по типам договоров.
  - Подсветка **отклонений от playbook** с градацией: критическое / среднее / приемлемое.
  - Предложение **redline** (предлагаемые правки) с обоснованием. Правки оформляются как track-changes в docx, чтобы юрист мог принять/отклонить.
  - Поиск **отсутствующих обязательных пунктов** (например, нет пункта о конфиденциальности).
  - Резюме договора в 5–10 пунктах простым языком — для бизнес-заказчика и для топ-менеджмента.
  - Оценка общего риск-скоринга (низкий/средний/высокий) с расшифровкой.
- Отчёт о проверке экспортируется в PDF и крепится к карточке договора.
- **Важно:** AI — ассистент, не финальное решение. Каждое предложение должно быть с источником (ссылка на пункт playbook), юрист принимает решение.

#### 4.2.4 Согласование (Approval workflow)

- Конфигурируемый маршрут согласования: последовательный, параллельный, условный (если сумма > X — добавить финдиректора, если иностранный контрагент — добавить compliance).
- Каждый согласующий видит карточку, может оставить комментарий, потребовать правки, согласовать.
- Версионирование: каждая итерация фиксируется, видно diff между версиями.
- Быстрые маршруты для типовых документов («короткий маршрут»).
- Уведомления в email + интеграция в Telegram/MS Teams/Slack по выбору.
- Дедлайны на каждом шаге с эскалацией при просрочке.

#### 4.2.5 Электронная подпись

- Интеграция с КриптоПро ЭЦП, СБИС, Контур.Диадок, DocuSign (если работа с иностранными контрагентами).
- Отправка контрагенту на подпись через защищённую ссылку.
- Сохранение подписанной версии с метаданными (кто, когда, IP, ОТУ).

#### 4.2.6 Пост-исполнение (Obligation management)

- Автоматический трекинг сроков: продление, расторжение, отчёты, оплата.
- Уведомления ответственному за 90/60/30/7 дней до критичных дат.
- Дашборд «Что у меня горит».
- Связка обязательств с фактами исполнения (через интеграцию с 1С/SAP — отметка о платеже или поставке).

### 4.3 МОДУЛЬ: Управление делами (Matter Management)

Зонтичный модуль для **всех нетиповых задач**: судебные дела, претензии, исполнительные производства, корпоративные процедуры, проверки контролирующих органов, расследования.

#### 4.3.1 Судебные и претензионные дела

- Карточка дела:
  - Реквизиты: номер дела, суд, инстанция, истец, ответчик, третьи лица, предмет спора, сумма иска, госпошлина, дата подачи, статус.
  - Календарь заседаний с напоминаниями.
  - Документы (иски, отзывы, доказательства, судебные акты).
  - Лента процессуальных событий.
  - Ответственный юрист + внешний консультант (если есть).
  - Бюджет дела (план/факт), почасовка внешних консультантов.
  - Прогноз исхода (AI-оценка на базе аналогичных дел из практики, опционально).
- **Интеграция с КАД Арбитр, ГАС Правосудие, Мосгорсуд, ФССП** — автоматическая синхронизация статусов дел и подгрузка процессуальных документов (как XSUD и Casebook).
- Реестр всех дел с фильтрами и группировками: по суду, инстанции, контрагенту, юристу, сумме, статусу.
- Дашборд: количество активных дел, сумма требований/риска, статистика выигранных/проигранных, нагрузка на юристов.

#### 4.3.2 Претензионная работа

- Реестр претензий (входящих и исходящих) со статусами.
- Шаблоны претензий и ответов на претензии.
- Связка с договором, по которому возникла претензия.
- Связка с дальнейшим судебным делом, если претензия не урегулирована.
- Дебиторка: интеграция с 1С/SAP по неоплатам → автоматическое формирование пред-претензионной работы.

#### 4.3.3 Исполнительное производство

- Реестр исполнительных листов, синхронизация с ФССП.
- Контроль взыскания, остаток долга, действия пристава.

#### 4.3.4 Корпоративные процедуры

- Реестр обществ группы (наши юрлица): реквизиты, учредительные документы, история изменений, бенефициары, состав СД, ГД.
- Календарь корпоративных событий: ОСА, СД, продление полномочий ГД, аудит.
- Шаблоны и автогенерация: протоколы ОСА, СД, решения единственного участника, уведомления.
- M&A-сделки: чек-листы due diligence, data room (защищённое хранилище для контрагента-покупателя), статусы.
- Реестр бенефициаров (анти-отмывочные требования, 115-ФЗ).

### 4.4 МОДУЛЬ: Контрагенты и due diligence

- Единый реестр контрагентов: данные из ЕГРЮЛ/ЕГРИП (автоподгрузка по ИНН), реквизиты, банковские счета, контактные лица, история взаимодействий.
- **Автоматическая проверка контрагента (compliance check)** при добавлении и периодически:
  - Действующее юрлицо? (ЕГРЮЛ)
  - Не банкрот? (Федресурс/ЕФРСБ)
  - Налоговая задолженность? (ФНС открытые данные)
  - Есть ли судебные дела? (КАД Арбитр)
  - Исполнительные производства? (ФССП)
  - В реестре недобросовестных поставщиков? (РНП ФАС)
  - В санкционных списках? (OFAC, ЕС, ООН — для иностранных)
  - Массовый учредитель/директор? (флаг ФНС)
- **Скоринг надёжности**: автоматическая оценка с расшифровкой, обновляется по расписанию.
- Подключение через Dadata, Контур.Фокус, СПАРК или аналог. Архитектура — провайдер-агностичная (адаптер).
- Карточка контрагента: история всех договоров, претензий, дел с этим контрагентом.

### 4.5 МОДУЛЬ: Доверенности

- Реестр выданных доверенностей: на кого, объём полномочий, срок действия, основание выдачи.
- Шаблоны доверенностей с автогенерацией.
- Реестр отозванных доверенностей.
- Уведомления об истечении.
- Маршрут подписания доверенности (вкл. нотариальные).
- Электронные доверенности (МЧД — машиночитаемая доверенность ФНС) — интеграция с распределённым реестром ФНС.

### 4.6 МОДУЛЬ: Compliance и мониторинг законодательства

- Подписка на изменения в законодательстве по релевантным темам (через API КонсультантПлюс/Гарант или парсинг pravo.gov.ru).
- AI-фильтрация: что из изменений касается компании (на базе профиля компании: отрасль, регионы, виды деятельности).
- Дайджест изменений еженедельно.
- Реестр обязательных требований к компании (отчётность в ФНС, Росреестр, Росприроднадзор, Ростехнадзор — критично для производства), календарь сроков сдачи.
- Реестр локальных нормативных актов (политики, регламенты, приказы) с владельцами и датами пересмотра.

## 5. Сквозные AI-возможности

AI должен быть **встроен в рабочий поток**, а не отдельным «AI-чат-окном в углу». Каждое применение должно давать **измеримую** экономию времени.

### 5.1 Извлечение данных из документов
Загрузил docx/pdf → AI вернул структурированный JSON с реквизитами, сторонами, суммами, сроками, ключевыми условиями. Используется в CLM, реестре договоров, делах.

### 5.2 AI-ревью договоров
См. п. 4.2.3. Это ключевая функция. Качество измеряется precision/recall на тестовом корпусе.

### 5.3 Резюме длинных документов
Любой документ (договор, иск, отзыв, протокол) → краткое резюме в 5–15 пунктов простым языком. С возможностью «копнуть» в каждый пункт (раскрытие).

### 5.4 RAG-поиск по корпоративной базе
Чат-помощник, отвечающий на вопросы юристов на базе **внутренних** договоров, дел, базы знаний. Только с источниками. Никаких галлюцинаций — если ответа нет в источниках, AI говорит «не нашёл».

### 5.5 Генерация черновиков
По заявке бизнес-заказчика и шаблону — AI готовит черновик документа (доверенность, претензия, ответ на претензию, простое соглашение). Юрист проверяет и дорабатывает.

### 5.6 Классификация и маршрутизация заявок
По тексту заявки определить тип, срочность, правильного исполнителя.

### 5.7 Прогноз исхода судебного дела (опционально, поздняя фаза)
На базе исторических данных компании и открытых данных судов.

### 5.8 Автоматическое извлечение обязательств
После подписания договора AI извлекает все обязательства (что, кто, когда) и создаёт записи в реестре обязательств с автоматическими дедлайнами.

### 5.9 Сравнение версий документов (semantic diff)
Не текстовый diff, а смысловой: «в новой версии увеличена неустойка с 0.1% до 0.3%».

**Принципы работы с AI:**
- Все AI-действия логируются.
- Для каждого AI-вывода — кнопка «не согласен» с обратной связью для дообучения.
- Никогда не отправлять конфиденциальные данные во внешние LLM без явной настройки (для критичных контуров — только локальная LLM).
- Предусмотреть конфиг «AI off» для отдельных типов документов с грифом.

## 6. Интеграции

Спроектируй систему как **открытую**, со стандартизированными адаптерами:

- **ЕГРЮЛ/ЕГРИП:** Dadata, ФНС API, Контур.Фокус, СПАРК (адаптер-провайдер).
- **Банкротство:** Федресурс/ЕФРСБ.
- **Суды:** КАД Арбитр, ГАС Правосудие, Мосгорсуд (через парсеры или агрегаторы типа Casebook API).
- **ФССП:** API исполнительных производств.
- **Электронная подпись:** КриптоПро, Контур.Диадок, СБИС, DocuSign.
- **ERP/учёт:** 1С, SAP, Microsoft Dynamics — через REST/SOAP/файловый обмен.
- **Документооборот:** Directum RX, ELMA, Docsvision (если уже есть в компании).
- **HRM:** для подгрузки сотрудников и оргструктуры.
- **Мессенджеры:** Telegram, Mattermost, MS Teams, Slack — для уведомлений.
- **Email:** SMTP + IMAP (опционально — приём заявок по email с парсингом).
- **Календари:** CalDAV, Google Calendar, Outlook — синхронизация заседаний и дедлайнов.
- **СЭД:** возможность пушить подписанные документы в корпоративную СЭД.
- **SSO:** SAML 2.0, OIDC, LDAP/AD.

Все интеграции — через **Adapter pattern**, чтобы можно было заменить провайдера без переписывания бизнес-логики.

Внутренний REST API + GraphQL для будущих интеграций. OpenAPI-спека генерится автоматически.

## 7. Безопасность

- **Хранение:** все документы шифруются at rest (AES-256), TLS 1.3 in transit.
- **Доступы:** RBAC + ABAC (контекстный — например, юрист видит только дела своего бизнес-юнита, если не указано иное).
- **Аудит:** каждое действие пользователя логируется. Audit log immutable (append-only, например, через PostgreSQL row-level триггеры или отдельный лог в WORM-хранилище).
- **MFA:** обязательно для всех ролей выше «бизнес-заказчика».
- **Защита от утечек:** водяные знаки на просмотр документов, запрет скачивания для определённых ролей, контроль печати (через DLP-метки).
- **Резервное копирование:** автоматическое ежедневное, с тестовым восстановлением раз в квартал.
- **Шифрование персональных данных:** в БД — на уровне колонок для критичных полей (152-ФЗ).
- **Хранение данных в РФ:** обязательно для российских пользователей (152-ФЗ).
- **Журнал согласия на обработку ПД** для бизнес-заказчиков и контрагентов.
- **Защищённые внешние порталы:** для внешних консультантов и контрагентов — изолированные доступы только к конкретным сущностям.
- **Penetration testing:** заложить процедуры регулярного аудита.
